## 1. Project Overview

IranRobot is a direct-sales, sourcing, and rental storefront for industrial and service robots. The business itself owns the inventory and the customer relationship — this is **not a marketplace**. There are no third-party sellers, no per-vendor catalogs, and no per-vendor payouts on the roadmap. Every product visible to the customer is curated and managed by IranRobot.

The UI is multilingual and RTL-capable. Persian (`fa`) is the default and Persian is treated as the primary copy on every record; English (`en`) is the secondary localization. The frontend mutates `document.documentElement.lang` and `dir` on language change (`fa` → `rtl`, `en` → `ltr`), and renders Jalali dates via `Intl.DateTimeFormat('fa-IR')` when the active language is Persian.

Commerce on the current site is **quote-first**:

- All 74 products in `src/data/robots.ts` have `priceUsd` and `rentPerDayUsd` unset.
- All 74 products carry `priceLabel = "استعلام قیمت"` / `priceLabelEn = "Request quote"`.
- All 74 products advertise `modes = ['procure']` only.

The `AvailabilityMode` type still defines `'buy' | 'rent' | 'procure'`, and the UI (PDP, RobotCard, QuoteDrawer, Rent page) is wired to handle all three. The backend must therefore continue to support buy and rent modes as first-class concepts, even though no current product exercises them.

The Frappe backend must provide, at minimum:

| Capability | Why it is needed |
|---|---|
| Bilingual product catalog (fa + en) with multi-axis filtering | Catalog sidebar filters by PLP id, subcategory, use-case, and `isNewArrival`. |
| Product Detail Page data (specs, gallery, related) | PDP reads full bilingual `specs`/`specsEn`, computes related-by-category. |
| Quote / Cart submission with mixed modes | `QuoteDrawer` holds `buy`, `rent`, and `procure` lines in one cart and distinguishes by `mode`. |
| Custom procurement requests | 3-step wizard submits `ProcurementRequest` (product, brand, qty, country, budget, notes, contact). |
| Rental requests | Currently a client-side calculator; needs a server-authoritative pricing/quote endpoint. |
| USD wallet with top-up and spend history | `walletBalanceUsd` + `WalletTx[]` are local today; must move server-side per user. |
| Support tickets | Support form is a 2.5s client-side mock today; no backend exists yet. |
| Per-user onboarding state | `onboardingSeen` flag is in `localStorage` today. |
| Multilingual content delivery (fa primary + en secondary) | Every bilingual field has a Persian primary and an `…En` sibling. |

## 2. Frontend Feature Summary

### 2.1 Homepage
File: home view composed from `ROBOTS[0]` plus 4 follow-up cards.

- Hero band with primary CTAs.
- Category mosaic driven by `src/data/categoryCarousel.ts` (see Section 4 for the homepage-only id set).
- Editor's Pick grid: featured robot = `ROBOTS[0]`, plus 4 more cards.
- CTA bands routing into `catalog`, `procurement`, `rent`, `finder`, `support`.

**Backend implication:** an endpoint (or cached payload) for "featured robot + N follow-ups" with full PDP-grade bilingual fields, since `RobotCard` reads `name`, `brand`, `origin`, `rating?`, `tagline`, `inStock`, `category`, `modes`, `priceLabel`/`priceUsd`. The featured-variant editorial section additionally reads `editorialBullets`, `bestFor`, and the first 4 entries of `specs` excluding SKU rows — populated on 1/74 today (`aimoga-mornine`).

### 2.2 Shop / Catalog
File: catalog view with sticky left sidebar.

- Sidebar renders the SHOP_TREE (see Section 4).
- `activeId` is driven by URL `route.param`, **not** hover. Hover-preview has been removed.
- Default `activeId` = first tree id whose matcher returns at least one product.
- Click-only filtering.

**Backend implication:** the backend must expose product counts per PLP id (so the sidebar can compute "first id with matches"), and must accept the PLP id namespace in list queries (not just the singular form-factor enum).

### 2.3 Product Detail Page (PDP)
File: `RobotDetail` component, slug-based lookup via `getRobotById`.

- Breadcrumb: Shop / category / product.
- Gallery: single image placeholder (`galleryCount=1`), but the `gallery: string[]` field exists (populated on 24/74, usually a single-element array equal to `image`).
- Star row: `rating.toFixed(1)/5` if `rating` is numeric, else "No reviews yet". `rating` is **never** populated in current data (0/74).
- Full bilingual specs table from `specs` / `specsEn`.
- Price block: numeric `priceUsd` renders via `usd()` + `tomanRange()`; otherwise `priceLabel` / `priceLabelEn`.
- Qty stepper: min 1, no max.
- Add to Cart: disabled with label "Quote only" when `modes` does not include `'buy'`.
- Add to Quote: full-width outline button, always `mode='procure'`.
- Related products: same `category`, exclude current, limit 4.

**Backend implication:** PDP responses must return full bilingual content plus `modes` and an explicit "is quote-only" affordance. Related-product computation can stay client-side or move server-side.

### 2.4 Quote Drawer (Cart Drawer)
File: `QuoteDrawer`.

This is a single surface for both purchase and quote items; the `mode` field on each `CartLine` distinguishes them.

- Each line shows `RobotIllustration` thumbnail, name, mode badge, qty stepper, line subtotal.
- Subtotal = `unit * qty` where `unit = rent ? (rentPerDayUsd ?? 0) * (days ?? 1) : (priceUsd ?? 0)`.
- Total = sum across lines.
- CTAs:
  - "Pay with wallet" when `walletBalanceUsd >= total` → calls `spendWallet(total, 'پرداخت سبد سفارش')`.
  - Otherwise "Top up" → closes drawer and routes to `wallet`.
  - "Clear" empties the cart.
- After successful pay: green check + amount.

**Backend implication:** because `procure` lines have `unit = 0` today, total for an all-quote cart is 0 and the wallet path is effectively disabled. Backend must support a "submit quote" path for `procure`-only carts that does not require wallet funds.

### 2.5 Procurement Wizard
File: `procurement` view, 3 steps.

- Step 1 (product details): `productName`, `brand`, `qty`.
- Step 2 (commercial): `country` from `COUNTRIES` (fa/en), `targetBudgetUsd?`, `notes`.
- Step 3 (contact): `contactName`, `contactPhone` (regex `/^[0-9۰-۹+\-\s]{7,}$/`, i.e. accepts both Western and Persian digits, length ≥ 7).
- Submission appends a `ProcurementRequest` with `status='pending'`.
- Status badge map: `pending` → Queued; `reviewing` → Reviewing; anything else → Quoted.

**Backend implication:** backend must accept Persian-digit phone numbers (normalize on ingest), persist country in both languages, and own the `pending → reviewing → quoted` lifecycle (frontend reads any non-pending/non-reviewing value as Quoted).

### 2.6 Rental Calculator
File: `rent` view, client-side only.

- Filters `ROBOTS` to entries with `modes.includes('rent')` AND a numeric `rentPerDayUsd`. **Today this list is empty** (0/74).
- Inputs: `selectedId`, `days` (1–365), `qty` (1–50), `withOperator`.
- Day-preset chips: 3, 7, 14, 30, 90.
- Operator surcharge: flat **$80/day**.
- Discount tiers: `days >= 30` → 15%, `days >= 14` → 8%, `days >= 7` → 4%.
- "Add to Cart" creates a line with `mode='rent'` and `days`.

**Backend implication:** the discount table, operator surcharge, and per-product daily rate must move to server-authoritative pricing so the cart subtotal can be trusted at checkout.

### 2.7 Robot Finder
File: `finder` view, 4 steps.

- Inputs: `useCase`, `budget`, `mode`, `urgency`.
- Local scoring:
  - +50 use-case match
  - +25 mode match
  - up to +20 budget match (closeness + `'flex'` bonus)
  - +15 `urgency='now'` AND `inStock`
  - +10 `urgency='month'` AND `leadTimeDays <= 35`
  - +5 `urgency='flex'`
  - `+ rating`
- Returns top 3.

**Backend implication:** scoring may stay client-side for now, but `useCases`, `modes`, `inStock`, `leadTimeDays`, `rating` must all be present in the list payload.

### 2.8 Wallet
File: `wallet` view.

- USD balance with Toman estimate (`USD_TO_TOMAN = 95000`, `tomanRange` ±6%, rounded to nearest 100,000 Toman).
- Top-up via `NumberInput` with preset chips: 100, 250, 500, 1000, 2500.
- `topupWallet(amt)` is a no-op when `amt <= 0`; otherwise prepends `{type:'topup', label:'افزایش موجودی کیف پول'}`.
- History list: txs sorted newest-first, colored by type (spend = brand red, others = emerald).
- **`spendWallet` is NOT called from the wallet view** — it is invoked by `QuoteDrawer`'s "Pay with wallet" button.
- The `WalletTx.type` union includes `'refund'`, but no current action emits a refund tx.

### 2.9 Support
File: `support` view.

- FAQ accordion.
- Contact form: `name`, `email` (`dir=ltr`), `topic` (Sales / Sourcing / Rental / Technical / Other), `message`.
- **Currently a client-side mock only**: on submit, `sent=true` for 2500ms, then resets. No backend call.

**Backend implication:** ticket DocType is greenfield; nothing on the frontend currently calls a server.

### 2.10 Multilingual / RTL
File: `src/i18n.tsx`, `src/lib/format.ts`.

- Languages: `'fa'` (default) and `'en'`. Persisted in `localStorage` under `iranrobot.v1.lang`.
- `t(fa, en)` picks one based on active lang — there is **no key-based dictionary**; the calling site supplies both strings.
- `n(v)` swaps to Persian digits when `fa`.
- `usd(v)` uses `Intl.NumberFormat('en-US', { style:'currency', currency:'USD', maximumFractionDigits:0 })`, then digit-swaps to Persian when `fa`.
- `date(ts)` / `dateTime(ts)` use `Intl.DateTimeFormat` with locale `'fa-IR'` (Jalali) when `fa`, `'en-US'` otherwise.
- Provider mutates `document.documentElement.lang` and `dir` on lang change.

**Backend implication:** since `t()` is call-site bilingual rather than key-based, the backend must return **both** `fa` and `en` values on the same record for every localized field (see Section 3 pairing table).

### 2.11 Onboarding
File: `Onboarding` modal component.

- 3-step modal, auto-opens on first visit when `!onboardingSeen`.
- `dismissOnboarding()` persists `seen=true` and resets to step 0.

### 2.12 localStorage Keys

All keys carry the prefix `iranrobot.v1.`:

| Key | Type | Source |
|---|---|---|
| `iranrobot.v1.cart` | `CartLine[]` | Cart state |
| `iranrobot.v1.wallet.balance` | `number` (USD) | Wallet balance |
| `iranrobot.v1.wallet.txs` | `WalletTx[]` | Wallet history |
| `iranrobot.v1.onboarding.seen` | `boolean` | Onboarding dismissal |
| `iranrobot.v1.procurement` | `ProcurementRequest[]` | Procurement queue |
| `iranrobot.v1.lang` | `'fa' \| 'en'` | Active language |

The `uid(prefix)` helper builds ids from a base36 timestamp plus 6 base36 random chars. It is **not cryptographically random**. Prefixes in use: `cl_` (cart lines), `tx_` (wallet txs), `pr_` (procurement requests). Backend ids should not collide with this scheme and should not rely on it for uniqueness.

### 4.7 Cross-namespace callouts

- **`solutions` is a special case.** It exists in the `RobotCategory` enum, as a virtual PLP entry, as a header dropdown group, and as a homepage mosaic id — but with **different semantics in each place**. The PLP uses `r.useCases.length > 0`. The enum value is unused. The homepage uses 8 marketing tiles that don't filter anything.
- **Unused enum values:** `cobots`, `mobile`, `educational`, `solutions` are valid `RobotCategory` values but have 0 products. The backend should keep them addressable; the catalog will simply render empty results.
- **Singular vs. plural drift:** `drone` (enum) vs. `drones` (PLP). The backend must accept either at the API boundary or expose an explicit mapping.
- **Legacy tag filter:** `regional` is a tag-based PLP (`r.tags.includes('regional')`) with no current matches; treat as deprecated unless the business intends to revive it.
- **Asset folder convention:** `public/assets/products/<category-or-mixed>/<sub-or-flat>/<slug>.webp`. Folder ids match SHOP_CATEGORIES ids and subcategory ids exactly. A `solutions/` asset folder exists (with `education`, `healthcare`, `inspection`, `security`, `warehouse` subfolders, plus an empty `custom/`) even though no product currently has `category='solutions'` — these assets back the use-case marketing tiles, not product images.

### Notes on the mapping

- Bilingual pairs are kept as paired flat fields on the same row (`name_fa` + `name_en`, `tagline_fa` + `tagline_en`, etc.) rather than a separate Translation table. This matches the frontend's `t(fa, en)` helper in `src/i18n.tsx`, which simply picks one of two values and is not key-based.
- `specs`, `highlights`, `editorial_bullets`, and `best_for` each collapse the `<field>`/`<field>En` pair into a single child row with `<col>_fa` and `<col>_en` columns. This guarantees the two arrays stay aligned (the frontend assumes `specs[i]` matches `specsEn[i]`) and removes the risk of mismatched lengths.
- `modes: AvailabilityMode[]` maps to three Check fields — `is_sellable` (`buy`), `is_rentable` (`rent`), `is_sourceable` (`procure`) — instead of a child table or Select. Three booleans are easier to filter on (PLP queries, Rent calculator filter `modes.includes('rent') && rentPerDayUsd`, RobotCard Add button mode resolution) and they map cleanly to website permission rules. The child-table alternative is acceptable but adds joins for every list query.
- `useCases` is a child table linking to `Robot Use Case`. A multi-Select would also work because the value set is small and stable (`education`, `warehouse`, `inspection`, `security`, `healthcare`, `custom`), but Link via a child table preserves referential integrity and lets the Solutions sub-PLP and Finder use a real Link filter.
- `priceUsd` is `Currency` with `options=USD`. Since 74/74 products have `priceUsd=null` today, the API must keep the field nullable and the PDP must continue to fall back to `price_label_fa`/`price_label_en`. The toman range shown on the PDP comes from `tomanRange(priceUsd)` in `src/lib/format.ts`; the backend stores USD only, FX conversion stays client-side.
- `priceLabel`/`priceLabelEn` are plain `Data` fields. They are not Select even though current data is uniform (`استعلام قیمت` / `Request quote`), because the frontend type marks them as free strings.
- `image` is `Attach Image`. Asset paths follow the folder convention `public/assets/products/<category-folder>/<sub-folder>/<slug>.webp` where the folder ids match `SHOP_CATEGORIES` ids exactly. Migration scripts can construct these from `category`, `subcategory`, and `slug`.
- `gallery` is a child table on `Robot Product` rather than a JSON column, so future multi-image PDPs (carousel, lightbox) can add `idx`, `alt_fa`, `alt_en`, and per-image metadata without altering the parent DocType. The first row should mirror `hero_image` for parity with today's single-image rendering.
- `rating`, `accent`, and `priceUsd` are nullable and never populated today; they are still included so the API contract matches the `Robot` interface 1:1 and the frontend does not need conditional shape handling.

## 23. API Response Standards

### 23.1 Success envelope

Every successful response — including empty lists — uses:

```json
{ "ok": true, "data": {}, "message": "Success" }
```

- `ok` is always the boolean `true`.
- `data` is always a JSON object. List endpoints wrap their array inside `data.items` (see 23.4) so the envelope shape stays stable.
- `message` is a short human-readable string suitable for toasts. Persian copy is allowed (e.g. `پرداخت سبد سفارش`), localization of `message` is the client's responsibility based on context.

### 23.2 Error envelope

```json
{ "ok": false, "error": { "code": "VALIDATION_ERROR", "message": "Quantity must be at least 1", "field": "qty" } }
```

- `ok` is always the boolean `false`.
- `error.code` is one of the standard codes in 23.3.
- `error.message` is a human-readable description. Single language at a time (en by default; fa when the request carries an `X-Lang: fa` header — TBD per section 24).
- `error.field` is optional; present for `VALIDATION_ERROR` so the client can highlight the right input.
- Additional, code-specific fields MAY be added under `error.details` without breaking the contract.

### 23.3 Standard error codes

| Code | When | HTTP status |
|------|------|-------------|
| `VALIDATION_ERROR` | Bad input — missing required field, regex mismatch, out-of-range int, unknown enum value. | 200 (whitelisted handler) |
| `NOT_FOUND` | Slug / id does not resolve, or resolves to a non-public record. | 200 |
| `AUTH_REQUIRED` | Caller is Guest but endpoint requires login (cart, wallet). | 401 |
| `FORBIDDEN` | Caller is logged in but lacks the required role (`Sales`, admin previews). | 403 |
| `RATE_LIMITED` | Public submission endpoints (`submit_quote_request`, `submit_procurement_request`, `submit_rental_request`, `submit_support_ticket`) exceeded their per-IP throttle. | 429 |
| `INTERNAL_ERROR` | Unhandled server exception. Stack trace logged server-side; message is a generic "Something went wrong" string. | 500 |

### 23.4 Pagination envelope

List endpoints (`get_products`, `get_wallet` history, future `list_quote_requests`, etc.) MUST wrap items and paging meta together:

```json
{
  "ok": true,
  "data": {
    "items": [],
    "pagination": {
      "page": 1,
      "page_size": 24,
      "total_items": 74,
      "total_pages": 4
    }
  },
  "message": "Success"
}
```

- `page` is 1-based.
- `total_pages = ceil(total_items / page_size)`; equals `0` when `total_items == 0`.
- Requesting a page past the last valid one returns `items: []` with the correct `total_*` values — NOT a `NOT_FOUND`.

### 23.5 HTTP status code mapping

Following Frappe's whitelisted-handler convention:

| Scenario | HTTP status | Body |
|----------|-------------|------|
| Successful handler return (including empty results, validation errors that are "expected business outcomes") | `200` | `ok: true` OR `ok: false` with `VALIDATION_ERROR` / `NOT_FOUND`. |
| Guest hitting an auth-required endpoint | `401` | `ok: false`, `AUTH_REQUIRED`. |
| Logged-in user missing role | `403` | `ok: false`, `FORBIDDEN`. |
| Rate limit exceeded | `429` | `ok: false`, `RATE_LIMITED`, include `Retry-After` header. |
| Unhandled server exception | `500` | `ok: false`, `INTERNAL_ERROR`. |

Clients should branch on `ok`, not on HTTP status, for everything in the 200-range. 401/403/429/500 are reserved for transport-layer concerns and should trigger generic error UI (re-login prompt, "try again later" toast).

Discount selection MUST use the highest tier whose threshold is satisfied — same ordering as the frontend conditional in `Rent.tsx`.

| Fieldname | Type | Notes |
|---|---|---|
| `naming_series` | Data | `RR-.YYYY.-.#####` |

### Server-side validation

- `end_date > start_date` (strict).
- `days = round((end_date - start_date) / 1d)`, then assert `1 <= days <= Rental Rate Config.max_days`.
- `1 <= qty <= Rental Rate Config.max_qty_per_request`.
- `robot.modes` MUST include `'rent'` AND `robot.rentPerDayUsd` MUST be set; otherwise reject with `400 robot_not_rentable`.

### Availability check

If insufficient stock is available for any day in the window, the transition fails with `409 insufficient_inventory` and the request stays in `Availability Check`.

| Reserved | Active | on `start_date` (scheduled job) |
| Active | Returned | on `end_date` or manual mark by Ops |
| Returned | Closed | after deposit reconciliation |

## 13. Cart and Order Flow

The cart is the same data structure that the Quote Drawer renders; see `CartLine` in `src/types.ts` and `addToCart`/`updateCartLine`/`removeCartLine`/`clearCart` in `src/context/AppContext.tsx`. The cart is the SINGLE channel that produces both Quote Requests (procure-only carts) and Sales Orders (buy/rent carts).

### Cart ownership

| Caller state | Cart key | Storage |
|---|---|---|
| Guest (no session) | anonymous client id, generated client-side and sent in a request header `X-Anonymous-Id`, OR Frappe `sid` cookie when one exists | `Guest Cart` DocType keyed by `session_id` |
| Logged-in Customer | `customer` Link | `Customer Cart` DocType keyed by `customer` |

On login, the backend MUST offer a "convert guest cart to customer cart" operation: take the rows from the Guest Cart matching `session_id`, merge them into the customer's cart applying the dedupe rule below, then delete the Guest Cart. This SHOULD be invoked automatically immediately after successful login.

### `Cart Item` shape

Mirrors `CartLine` from `src/types.ts`:

| Fieldname | Type | Notes |
|---|---|---|
| `robot_slug` | Link | Robot Product |
| `mode` | Select | `buy,rent,procure` |
| `qty` | Int | min 1 |
| `days` | Int | optional, only when `mode='rent'` |
| `notes` | Small Text | optional |
| `added_at` | Datetime | server time; frontend's `addedAt` (ms since epoch) is informational only |

### Dedupe rule

Backend MUST replicate the frontend dedupe in `addToCart` (see `src/context/AppContext.tsx`). When an `add_to_cart` call hits an existing line with the same `(robot_slug, mode)` tuple:

1. Add the incoming `qty` to the existing line's `qty`.
2. If `days` was provided in the patch, overwrite the existing `days`.
3. If `notes` was provided in the patch, overwrite the existing `notes`.
4. Do NOT create a new line, do NOT change `added_at`.

A different `(robot_slug, mode)` combination always creates a new line — buy and procure for the same robot coexist.

### Cart -> downstream document mapping

| Cart composition | Checkout produces |
|---|---|
| All lines `mode='procure'` | `Quote Request` (see section 10) |
| Any line `mode='buy'` with `priceUsd` set | `Sales Order` |
| Any line `mode='rent'` | `Rental Request` per distinct `robot` (see section 12) |
| Mixed buy + procure | `Sales Order` for buy lines AND `Quote Request` for procure lines, linked via `created_from_cart_id` |

Current data note: zero products have `priceUsd`, so the Sales Order path is effectively dormant today.

### Order status workflow (Sales Order)

`Draft -> Awaiting Payment -> Paid -> Confirmed -> Shipped -> Delivered -> Closed | Refunded`

| From | To | Trigger |
|---|---|---|
| Draft | Awaiting Payment | customer confirms checkout |
| Awaiting Payment | Paid | wallet debit success OR gateway success |
| Paid | Confirmed | Ops review |
| Confirmed | Shipped | Ops marks shipment |
| Shipped | Delivered | carrier confirmation |
| Delivered | Closed | system, after return window |
| any (Paid+) | Refunded | Ops issues refund; emits wallet credit (see section 14) |

Payment gateway selection for fixed-price orders is TBD — see Open Questions.

### Wallet payment path

The QuoteDrawer "Pay with wallet" CTA invokes `spendWallet(total, 'پرداخت سبد سفارش')` from `src/context/AppContext.tsx`. The button is only rendered when `walletBalanceUsd >= total`; otherwise the drawer shows a "Top up" button that closes the drawer and routes to `#/wallet` via `go('wallet')`.

Backend mirror: a single transactional endpoint that, in one DB transaction:

1. Recomputes the order total server-side (do NOT trust the client value).
2. Locks the customer's `Wallet` row.
3. Asserts `balance_usd >= total_usd`; if not, returns `409 insufficient_funds`.
4. Decrements `balance_usd` by `total_usd`.
5. Inserts a `Wallet Transaction` with `type='spend'`, `label_fa='پرداخت سبد سفارش'`, `label_en='Cart payment'`, `reference_doctype='Sales Order'`, `reference_name=<order>`.
6. Transitions the Sales Order from `Awaiting Payment` to `Paid`.

All six steps either commit together or roll back together.

## 14. Wallet Requirements

The wallet is the single in-house payment instrument and the only checkout method currently wired up on the frontend. There is exactly one canonical currency.

### Currency policy

- All balances and transaction amounts are stored in USD as `Currency` fields.
- Toman amounts are presentation-only. The conversion (`USD_TO_TOMAN = 95000`) and the `+/-6%` spread in `tomanRange` live in `src/lib/format.ts` and remain client-side. The backend MUST NOT persist Toman amounts.
- Persian-digit rendering of numbers is also client-side (`src/lib/format.ts` `toFa`, `n`, `usd`).

### `Wallet` DocType (1:1 with Customer)

| Fieldname | Type | Notes |
|---|---|---|
| `customer` | Link | Customer; `unique=1` |
| `balance_usd` | Currency | non-negative; server-enforced |
| `last_tx_at` | Datetime | denormalized for sort/filter convenience |

### `Wallet Transaction` DocType

| Fieldname | Type | Notes |
|---|---|---|
| `naming_series` | Data | `WT-.YYYY.-.#####` |
| `wallet` | Link | Wallet |
| `amount_usd` | Currency | signed: positive for `topup`/`refund`, negative for `spend` |
| `type` | Select | `topup,spend,refund` — mirrors `WalletTx['type']` from `src/types.ts` |
| `label_fa` | Data | Persian human label |
| `label_en` | Data | English human label |
| `reference_doctype` | Link | DocType, e.g. `Sales Order`, `Quote Request`, `Top Up` |
| `reference_name` | Dynamic Link | referenced record |
| `created_at` | Datetime | server time; the frontend's `at` (ms epoch) is informational |
| `status` | Select | `pending,confirmed,failed` (top-ups remain `pending` until gateway success) |

### `topupWallet` API

Mirrors the frontend's `topupWallet(amt)` in `src/context/AppContext.tsx`, which no-ops for `amt <= 0` and prepends a transaction with `type='topup'` and `label='افزایش موجودی کیف پول'`.

Backend behavior:

1. Reject `amount_usd <= 0` with `400 invalid_amount`.
2. Create a `Wallet Transaction` with `type='topup'`, `status='pending'`, `label_fa='افزایش موجودی کیف پول'`, `label_en='Wallet top-up'`.
3. Hand the customer off to the payment gateway (TBD — see Open Questions).
4. On gateway success webhook: in a single transaction, set `status='confirmed'`, increment `Wallet.balance_usd` by `amount_usd`, update `Wallet.last_tx_at`.
5. On gateway failure or timeout: set `status='failed'` and DO NOT touch the balance.

### `spendWallet` API

Mirrors `spendWallet(amt, label)` in `src/context/AppContext.tsx`, which returns a boolean (true on success). The boolean contract surfaces in the QuoteDrawer's "Pay with wallet" success/fail rendering.

Backend behavior — atomic, single DB transaction:

1. Reject `amount_usd <= 0` (the frontend treats this as auto-success; the backend MUST reject to keep ledger integrity).
2. Lock `Wallet` row.
3. If `balance_usd < amount_usd`, return `409 insufficient_funds`.
4. Insert `Wallet Transaction` with `type='spend'`, `amount_usd = -amount`, `label_fa`/`label_en` supplied by caller, `reference_doctype`/`reference_name` supplied by caller, `status='confirmed'`.
5. Decrement `Wallet.balance_usd` by `amount`.
6. Update `Wallet.last_tx_at`.

Response shape:

```json
{
  "ok": true,
  "transaction": "WT-2026-00042",
  "balance_usd": 1240
}
```

On `insufficient_funds`:

```json
{
  "ok": false,
  "error": "insufficient_funds",
  "balance_usd": 30,
  "required_usd": 175
}
```

### Refunds

The frontend's `WalletTx['type']` union includes `'refund'` but no client action emits it — refunds are server-originated. The order workflow (section 13) MUST emit a refund transaction when a `Sales Order` transitions to `Refunded`:

- `type='refund'`
- `amount_usd` positive (credits the wallet)
- `label_fa` / `label_en` carry the refund reason (backend is the source of truth for refund labels — there is no seeded pair to mirror)
- `reference_doctype='Sales Order'`, `reference_name=<order>`
- `status='confirmed'` on creation
- `Wallet.balance_usd` incremented in the same DB transaction

### History endpoint

The Wallet view (`src/views/Wallet.tsx`) renders txs in descending recency, color-coded (`spend` red, others emerald). The backend list endpoint MUST:

- Return rows ordered by `created_at DESC`.
- Include `type`, `amount_usd` (signed), `label_fa`, `label_en`, `created_at`, `reference_doctype`, `reference_name`.
- Support paging (default 50, max 200).

### Seeded labels

Two existing frontend strings MUST be seeded as the canonical (fa, en) pairs for the two operations the client currently triggers; the backend re-emits these exact strings so the Wallet history is consistent regardless of source:

| Action | `label_fa` | `label_en` |
|---|---|---|
| `topupWallet` (client-initiated top-up) | `افزایش موجودی کیف پول` | `Wallet top-up` |
| `spendWallet` from QuoteDrawer "Pay with wallet" | `پرداخت سبد سفارش` | `Cart payment` |

For any other transaction type or future label, the backend is the source of truth — the client renders whichever of `label_fa` / `label_en` matches the active `iranrobot.v1.lang` value.

### Relevant frontend files

- `src/views/Wallet.tsx` — top-up form, preset chips (100/250/500/1000/2500), history list.
- `src/components/QuoteDrawer.tsx` — "Pay with wallet" / "Top up" CTAs, post-pay success state.
- `src/context/AppContext.tsx` — `topupWallet`, `spendWallet`, `walletBalanceUsd`, `walletTxs`.
- `src/lib/format.ts` — `usd`, `tomanRange`, `USD_TO_TOMAN`, `toFa`, `n` (presentation-only).
- `src/types.ts` — `WalletTx` union and shape.

## 15. Authentication and Users

The storefront is a hash-routed SPA with no login UI today. The backend must therefore support a fully anonymous browsing and lead-capture flow, while also providing a forward-compatible Customer account model that the SPA can adopt later.

### 15.1 Guest behavior

Most public endpoints are Guest-OK and must work without a session user. Concretely, a guest can:

- Browse the catalog (PLP) and category tree.
- Open a product PDP (`#/robot/<slug>`).
- Submit a Support ticket from `Support.tsx`.
- Submit a Quote Request from `QuoteDrawer` (the cart drawer).
- Submit a Procurement Request from the 3-step wizard in `Procurement.tsx`.
- Submit a Rental Request from `Rent.tsx`.
- Manage a session-scoped cart. The current frontend persists this in `localStorage` under `iranrobot.v1.cart`; the backend MUST accept guest cart ops keyed by an anonymous `cart_token` cookie so that server-side cart state is possible later without breaking the existing client.

Guests do NOT get a wallet, do NOT see request history, and cannot top up. The wallet view (`Wallet.tsx`) is only meaningful for an authenticated Customer.

- Address book (Frappe stock `Address` linked via Dynamic Link to Customer — TBD, see Open Questions).

- Optional: per-customer **API key + API secret** (Frappe's built-in) for SSR rendering or external integrations, sent via `Authorization: token <key>:<secret>`.
- Login UI in the SPA is TBD — see Open Questions.

## 16. Multilingual and RTL Requirements

### 16.1 Translation model

The frontend does NOT use a key-based dictionary. `src/i18n.tsx` exposes `t(fa, en)` which picks one of the two strings at the call site based on the active language. Implication: **the backend must return both `fa` and `en` variants for every localizable field on every response.** The client picks via `lang`.

### 16.2 Languages and persistence

| Item | Value |
|---|---|
| Supported languages | `fa` (default), `en` |
| Persisted on client | `localStorage` key `iranrobot.v1.lang` |
| Persisted on server | Not required for guests. For Customers, optional `preferred_language` field on `Customer` — TBD, see Open Questions |
| Direction | `rtl` for `fa`, `ltr` for `en`. Set client-side by `I18nProvider` on `document.documentElement`. Backend has no RTL responsibility |

### 16.3 Field pairing convention

Every localized string field appears as a paired set. The Persian field is primary (matching the source-of-truth in `src/data/robots.ts`).

| Frontend (`Robot`) | API response (`get_product_detail`) |
|---|---|
| `name` / `nameEn` | `name_fa` / `name_en` |
| `tagline` / `taglineEn` | `tagline_fa` / `tagline_en` |
| `description` / `descriptionEn` | `description_fa` / `description_en` |
| `origin` / `originEn` | `origin_fa` / `origin_en` |
| `priceLabel` / `priceLabelEn` | `price_label_fa` / `price_label_en` |
| `specs[]` / `specsEn[]` | `specs[]` with `label_fa`, `label_en`, `value_fa`, `value_en` |
| `highlights[]` / `highlightsEn[]` | `highlights_fa[]`, `highlights_en[]` |
| `editorialBullets[]` / `editorialBulletsEn[]` | `editorial_bullets_fa[]`, `editorial_bullets_en[]` |
| `bestFor[]` / `bestForEn[]` | `best_for_fa[]`, `best_for_en[]` |

Notes:

- `tags[]` and `useCases[]` are NOT localized (lowercase identifiers). Their human-readable labels come from the `Robot Use Case` DocType, which itself has `label_fa`/`label_en`.
- `priceLabel` is always the literal Persian string `استعلام قیمت` and `priceLabelEn` is always `Request quote` in the current data. The backend must allow free-form values (do not hardcode).

### 16.4 Fallback rule

When one of a localized pair is empty, return the other as-is. The backend MUST NOT silently copy `name_fa` into `name_en` — empty means empty, and the client decides whether to fall back. This preserves the editorial intent (e.g. some `highlights_en[]` arrays are deliberately empty today).

### 16.5 Numbers, currency, dates

| Concern | Frontend behavior | Backend responsibility |
|---|---|---|
| Digits | `toFa` swaps `0-9` -> Persian digits when `lang==='fa'` | Always return ASCII digits in JSON |
| Currency | `usd(v)` uses `Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' })` then digit-swaps | Always return raw numeric USD (no formatting) |
| Toman estimate | `tomanRange(priceUsd)` uses `USD_TO_TOMAN = 95000`, +/- 6% rounded to 100,000 Toman | TBD: expose `USD_TO_TOMAN` via a `Currency Settings` Single DocType so it can be tuned without a redeploy — see Open Questions |
| Dates | `date(ts)` / `dateTime(ts)` use `Intl.DateTimeFormat('fa-IR')` (Jalali) when `fa`, `'en-US'` otherwise | Always return ISO 8601 strings or epoch milliseconds. NO Jalali conversion on the server |

### 16.6 Persian digits in input

The phone validation regex `^[0-9۰-۹+\-\s]{7,}$` accepts both Western and Persian digits. The backend MUST mirror this regex and normalize Persian digits to ASCII before persisting `Procurement Request.contact_phone` and analogous fields. Document this normalization in section 20.

## 21. Security Requirements

### 21.1 Guest-safe public APIs

Mark these `@frappe.whitelist(allow_guest=True)`:

- `get_categories`, `get_use_cases`, `get_featured_product`, `get_products`, `get_product_detail`, `get_carousel`.
- `get_cart` / `add_to_cart` / `update_cart_item` / `remove_cart_item` / `clear_cart` — accept `cart_token` cookie for anonymous use.
- `submit_quote_request`, `submit_procurement_request`, `submit_rental_request`, `submit_support_ticket`.

Everything else (wallet, request history, admin) requires an authenticated session.

### 21.2 Rate limiting

Apply per-IP limits on the four `submit_*` endpoints. Suggested defaults:

| Endpoint | Per minute | Per day |
|---|---|---|
| `submit_quote_request` | 5 | 50 |
| `submit_procurement_request` | 5 | 50 |
| `submit_rental_request` | 5 | 50 |
| `submit_support_ticket` | 5 | 50 |

Implement via Frappe's request hooks plus a `Rate Limit Log` DocType (or `frappe.rate_limiter`). Return HTTP 429 with a `Retry-After` header on breach.

### 21.3 Captcha

Add hCaptcha or Cloudflare Turnstile (TBD — see Open Questions) on the same four `submit_*` endpoints. Token is sent in the JSON body as `captcha_token`; the server verifies against the vendor and rejects on failure (HTTP 400).

### 21.4 Internal vs. public fields

The following fields MUST be marked `Permlevel: 1` and stripped from any Guest / Customer response:

- `Quote Request`: `supplier_name`, `supplier_quote_usd`, `supplier_lead_time_days`, `internal_notes`, `assigned_to`, `margin_pct`.
- `Procurement Request`: `supplier_name`, `supplier_quote_usd`, `internal_notes`, `assigned_to`.
- `Rental Request`: `internal_notes`, `assigned_to`, `assigned_unit` (Link to `Rental Inventory`).
- `Robot Order`: cost price, supplier link, internal notes.

Public `get_*` endpoints whitelist only the safe fields explicitly — never `frappe.get_doc().as_dict()` raw.

### 21.5 Role-based access

Enforced via the permission matrix in section 15.4. In addition:

- Customer endpoints (`get_my_requests`, `get_wallet`, etc.) filter by `frappe.session.user` server-side. Never accept a `customer` parameter from the client.
- Guest cart access is scoped by `cart_token` (HTTP-only cookie). Cart endpoints reject requests where the requested cart's token does not match.

### 21.6 CSRF

Default Frappe behavior: enforce CSRF on all POST/PUT/DELETE. The SPA sends `X-Frappe-CSRF-Token` on every state-changing request. Do not disable CSRF for any whitelisted endpoint.

### 21.7 Wallet spending

`spend_wallet(amount_usd, label)` MUST:

1. Run inside a single DB transaction.
2. Re-fetch the customer's `Wallet.balance_usd` with `SELECT ... FOR UPDATE` (row lock).
3. Reject if `amount_usd <= 0` (no-op, returns `false`).
4. Reject if `balance_usd < amount_usd` (returns `false`).
5. Otherwise insert a `Wallet Transaction { type: 'spend', amount_usd, label, customer }` and decrement `balance_usd`.
6. Commit, return `true`.

Never trust a client-computed balance or total. The frontend `QuoteDrawer` computes `total` locally but the server MUST re-compute from current cart contents and current product prices before charging.

### 21.8 Supplier and pricing leakage

The public `Quote Request` response returned to the submitter contains only: `id`, `status`, `created_at`, the items they submitted, and the public `price_label_*`. It MUST NOT include `supplier_quote_usd` or any internal margin until and unless the Sales User explicitly publishes a customer-facing quote (separate field, e.g. `customer_quote_usd`, gated by `status='Quoted'`).

### 21.9 Audit logging

Write a Frappe `Activity Log` (or custom `Lead Audit Log`) entry for:

- Every `submit_quote_request`, `submit_procurement_request`, `submit_rental_request`, `submit_support_ticket` (record IP, user agent, cart_token / user).
- Every `topup_wallet` and `spend_wallet` (record amount, resulting balance).
- Every manual wallet adjustment from Desk (record admin user, reason).
- Every status change on Quote / Procurement / Rental requests.

## 24. Open Questions for Product Owner

- **Quote-only is the universal default today.** All 74 products use `priceLabel='استعلام قیمت'` and `modes=['procure']`. Will any product ever ship with a fixed `priceUsd` or a daily `rentPerDayUsd`? If yes, which categories / SKUs first?

- **Refund transactions.** `WalletTx.type` includes `refund` but no current action emits one. Confirm: are refunds only issued by admins via the Desk, or also automatically on order cancellation?

### Payments and checkout

- **Payment gateway** for wallet top-up and order checkout: Iranian gateway (ZarinPal, IDPay, Pay.ir), foreign card (Stripe / PayPal), IBAN bank transfer, or a mix? This drives the Phase 3 integration.
